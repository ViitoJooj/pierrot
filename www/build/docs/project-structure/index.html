Project structure

Where files go, how folders become routes, and the shared anatomy of every .pierrot — page, component or layout.

Folder layout

The paths are not magic: the entry point is app.entry from settings.pierrot.json (default ./src/main.pierrot), and its folder becomes the project's src root. Everything else is relative to that folder.

→ See configuration to change these paths.

Routes = folders in pages/

Each pages/<route>/index.pierrot becomes a route. The folder name is the path:

FolderRoute
pages/home/index.pierrotdepends on set.Default (usually /)
pages/about/index.pierrot/about
pages/blog/post/index.pierrot/blog/post

The / route points to the page marked with set.Default(...) in the layout. Non-existent routes fall back to the set.Fallback(...) page with status 404.

Anatomy of a .pierrot file

Every .pierrot — page, component or layout — has the same shape: an optional <script> followed by the template.

What the parser does with each part:

In <script>BecomesDetail
import "x.css";stylesheetadded to <head> (dev) or to the bundle (build)
import "x.ts";script chunkbundled into the page
import { Name } from "x.pierrot";componentusable as <Name /> in the template
set.X(...)metadatatitle, icon, default, fallback…
let name: type; (no value)propdeclares a component prop
any other codelogictransformed TS → JS and sent to the browser

The import, set.X and prop-declaration lines are stripped from the script that ships to the browser — they are instructions for the compiler, not runtime code.

The three roles of a .pierrot

  • Layout (main.pierrot): the entry. Has <Slot /> (where the page is injected) and the global set.X. May wrap the page with components.
  • Page (pages/*/index.pierrot): the content of a route. May override the layout's metadata.
  • Component (components/*/index.pierrot): a reusable piece, with props.
