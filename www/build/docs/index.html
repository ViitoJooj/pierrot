Pierrot documentation

Pierrot is a single-file component framework compiled in Go. The CLI understands the .pierrot language, compiles everything to static HTML and ships the browser a runtime of a few kilobytes — no virtual DOM, no eval, no node_modules.

This is the full documentation. If you just want to see something running, jump to Hello World in 30s.

What Pierrot is

You write the UI in .pierrot files. Each file is a single component that bundles three things — an optional <script> (imports, metadata, props and TypeScript logic) and the template (HTML plus directives):

The server (pierrot dev) or the build (pierrot build) reads this file, resolves imports and components, turns the TypeScript into JavaScript with esbuild and assembles a complete HTML document. The browser gets ready-made HTML plus a small runtime that re-evaluates the dynamic bits when an event changes the state.

There is no JS build step in your project, no package.json, no virtual DOM. State is just the top-level variables of the <script>; every @event calls __pierrotUpdate() at the end to repaint the affected spans and blocks.

Architecture

The whole compiler lives in internal/. It is a short pipeline of pure functions that runs per request under pierrot dev and once per page under pierrot build. No giant AST, no worker pool, no cache.

#StepFileWhat it does
1parsereaders/parser.goSplits the <script> from the template. Extracts CSS/TS/component imports, set.X metadata and props.
2expansionworkers/dev_server.goRecursive: layout → page → components. <Slot /> receives the page; each <Name /> receives the component HTML, props applied per instance.
3templateworkers/template.go@for/@if blocks become JS functions. @render becomes a placeholder. // comments are dropped.
4bindingsworkers/dev_server.goPlain ${var} becomes <span data-bind>. @bind becomes oninput. @event becomes on<event> + update.
5transformesbuildEach <script> and each import "*.ts" becomes a TS → JS chunk. get.Dotenv("X") is replaced by the literal value first.
6assemblyworkers/dev_server.goFinal HTML: <head> with title/meta/CSS links, <body> with the markup, and the generated runtime.

The browser runtime

The runtime uses no eval. The server already knows the names used in ${...} and generates a state object from them. On every __pierrotUpdate():

  • data-bind spans get the variable's value;
  • data-pierrot-block blocks (@for/@if) re-run their JS function and re-render;
  • data-pierrot-expr spans (composite expressions) are re-evaluated;
  • inputs with @bind get the variable's value (skipping the focused element);
  • @render placeholders are refilled (html / markdown / preview iframe).

Quick conventions

  • Routes = folders in src/pages/. pages/about/index.pierrot → route /about.
  • The default page (/) and fallback (404) are set in the layout with set.Default(...) and set.Fallback(...).
  • Components start with an uppercase letter (<Header />); lowercase tags are plain HTML.
  • Props are declared in the component as let name: type; with no value.
  • Assets (images, fonts, robots.txt, favicon.ico) live in src/assets/.
